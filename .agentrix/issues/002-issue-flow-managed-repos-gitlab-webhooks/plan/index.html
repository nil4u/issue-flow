Issue Flow · 内部部署管理台 · GitLab Webhook Gateway

把 issue-flow 从“安装脚本 + Agentrix bridge”升级为可独立接 GitLab Webhook 的内部服务

核心变化：GitLab webhook 直接打到 issue-flow；仓库 token、webhook secret、安装状态和自动化策略在 issue-flow 管理页配置；Agentrix 只保留 agent task runtime 依赖，不再承接 GitLab webhook。

需评审的关键决策 优先 server-side dispatch

不再把每个事件转换成 GitLab CI pipeline trigger。CI bridge 只作为兼容模式，否则“token 配在 issue-flow”无法闭环。

当前链路：Agentrix daemon webhook bridge 是必经入口

  1. GitLab webhook项目 webhook 指向 Agentrix/daemon bridge
  2. @xmz-ai/gitlab-webhook-bridge规范化事件，触发 GitLab pipeline
  3. GitLab CI job读取 GITLAB_BRIDGE_* / AGENTRIX_* 变量
  4. issue-flow dispatch.cjs在 CI 里运行，需要仓库 CI 变量中的 token
  5. Agentrix task通过 agentrix-run 创建/恢复任务
问题：安装复杂；每个仓要配 CI 文件和变量；webhook 入口依赖 Agentrix；GitLab token 不在 issue-flow 自己的配置域内。

目标链路：issue-flow 自己承接 webhook 与仓库配置

  1. 管理页添加仓库保存 GitLab token、webhook secret、Agentrix API 配置
  2. GitLab webhook → issue-flowURL 指向 /webhooks/gitlab/{repoId}
  3. Webhook Gateway校验、去重、记录 delivery、规范化事件
  4. Server-side Dispatcher直接调用 issue-flow runtime/provider adapter
  5. GitLab API + Agentrix API用服务保存的凭据更新 issue/MR/label，并创建或恢复任务
收益：接入只需“添加仓库 + 配 webhook”；token 在 issue-flow 加密存储；Agentrix webhook bridge 退出主链路。

正常自动化路径

GitLab webhook 进入 issue-flow Gateway，校验 secret 后记录 delivery；Normalizer 产生 issue-flow event context；Dispatcher 使用保存的仓库 token 调 GitLab API，并在需要时调用 Agentrix 创建/恢复任务。

管理页应该让接入变成“添加仓库 → 复制 webhook → 验证通过”

这不是单纯 UI，页面上的每个控件都对应一个后端配置对象、一次校验或一个可诊断状态。

1选择 GitLabbaseUrl/apiUrl/project path
2配置 Token校验 /user、project、labels、files 权限
3配置 AgentrixbaseUrl/apiKey/runner/agent
4选择安装模式server-side 或兼容 CI bridge
5生成 WebhookURL + secret + event list
6发送测试事件delivery/run 可见，失败可重试

仓库详情页状态

  • Installed：runtime 文件版本与 manifest 匹配
  • Webhook OK：最近测试事件通过校验并完成路由
  • Token OK：最近校验成功，显示 scopes/username/fingerprint
  • Automation：auto default、review enabled、默认 agent

必须避免的 UI 泄漏

  • 不回显完整 token / secret
  • 错误详情不包含 provider token 或 Authorization header
  • 重放 delivery 时只展示 sanitized payload 摘要
  • Agentrix task env 不注入 GitLab provider token

Webhook 生命周期：把 Agentrix 的能力拆成 issue-flow 自己需要的四层

不是把 Agentrix 的表和 API 整体搬过来，而是迁移关键行为：校验、delivery 去重、GitLab event normalization、route side effects。

01

Receive

读取 headers、raw body、GitLab event type、delivery id，按 repoId 找配置。

02

Verify

对比 `X-Gitlab-Token` 和仓库 webhook secret；失败直接 401，记录安全事件。

03

Dedupe

按 provider/repo/deliveryId/consumer 去重；无 delivery id 时用 canonical payload hash。

04

Normalize

使用 `@xmz-ai/gitlab-webhook-bridge` 或抽出的 normalizer 生成 GitHub-style events。

05

Route

映射到 issue-flow actions：auto、comment、review、review-comment、pr-merged、pipeline-failed。

06

Dispatch

用保存的 token 调 GitLab API；用 Agentrix API 创建/恢复 task；写 DispatchRun。

建议数据模型切片

保持 issue-flow 服务自己的轻模型，吸收 Agentrix 的设计经验但不继承其完整平台模型。

GitProviderConfig

id, type, baseUrl, apiUrl, name

对应 Agentrix `GitServer` 的轻量版本。

RepositoryConfig

id, providerId, projectId, fullName, defaultBranch, installMode

仓库配置与 webhook URL 的主实体。

RepositoryCredential

repoId, encryptedToken, fingerprint, username, scopes

对应 Agentrix `LocalPatGitServerCredential` 的加密边界。

WebhookSecret

repoId, encryptedSecret, lastRotatedAt

只生成/重置，不明文常驻前端。

AutomationConfig

autoDefault, reviewEnabled, agent, runnerId

替代散落在 CI variables 的配置。

WebhookDelivery / DispatchRun

deliveryId, event, status, action, sanitizedError

诊断、重放和幂等的基础。

实施拆分

先把 webhook 和 token 域从 Agentrix bridge 中摘出来，再做安装体验和完整诊断。

P0

代码重构准备

把 issue-flow runtime provider/dispatch 抽成可被服务端调用的模块；保留 CLI 入口。

验收：现有 test 全过，CLI 行为不变。
P1

Repo Config + Credential

新增服务端、DB、加密 token、GitLab project 校验、仓库列表/详情 API。

验收：能添加仓库并校验 token。
P2

GitLab Webhook Gateway

实现 secret 校验、delivery 去重、normalization、action routing、dispatch run log。

验收：直接 webhook 跑通 issue/comment/MR/pipeline failure。
P3

管理页面与安装管理

添加仓库向导、webhook 配置页、runtime 安装/升级、label sync、诊断/重放。

验收：无需手工 curl 安装即可接入一个仓。
P4

迁移与兼容

保留旧 CI bridge 模式，提供逐仓迁移状态和回滚路径。

验收：旧仓不坏，新仓不依赖 Agentrix bridge。
